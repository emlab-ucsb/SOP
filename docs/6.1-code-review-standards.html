emLab Standard Operating Procedures

6.1 Code Review Standards

emLab is committed to producing replicable, transparent, and high-quality code. The following standards build on the best practices described in the Code chapter of this SOP and establish a consistent review process across all projects.

6.1.1 Documentation Standards

All code should be documented well enough that a colleague unfamiliar with the project can understand what the code does, why it does it, and how to run it. At a minimum, every project repository should include:

  • A README.md that describes the project, its file structure, and instructions for reproducing the analysis (see the Coding Pipelines section for more detail).
  • Script-level headers that describe the purpose of each script, its inputs, and its outputs (see the Scripts and Version Control section for a template).
  • Inline comments where the logic is not self-evident. Code should be self-documenting as much as possible — use clear variable and function names before resorting to comments.
  • A record of package dependencies managed with a tool like renv (see the Package Management section).

6.1.2 Self-Review Process

Before requesting a peer review, the original coder should conduct a self-review. A self-review is a deliberate, structured pass through your own code with fresh eyes. The goal is to catch issues early and respect your reviewer’s time. A self-review should include, at minimum:

  • Re-read the code. Step through the code as if you are seeing it for the first time. Check that the logic is correct and that the code does what you intend.
  • Run the full pipeline. Confirm that the code runs end-to-end without errors in a clean environment (e.g., restart your R session before running).
  • Check outputs. Verify that results, figures, and tables are reasonable and match expectations.
  • Review documentation. Ensure the README, script headers, and inline comments are accurate and up to date.
  • Clean up. Remove dead code, debugging artifacts, and commented-out experiments that are no longer needed.

6.1.3 Peer Review Process

Peer review is a core practice at emLab. As described in the Internal Code Review section, every project with at least two researchers should include a systematic review of major coding elements by a researcher who did not write the code.

The following guidelines supplement the existing SOP guidance:

  • Scope the review. The original coder should clearly communicate what needs to be reviewed — whether it is the full pipeline or a specific analysis. Use the coding pipeline documentation (e.g., README, Makefile, or targets pipeline) to guide the reviewer to the critical code.
  • Use pull requests when possible. Pull requests on GitHub provide a structured forum for line-by-line feedback, threaded discussion, and a permanent record of the review. For projects that do not use a branching workflow, the reviewer and coder should agree on an alternative method (e.g., a shared document or email).
  • Focus on substance. Reviews should prioritize correctness, reproducibility, and clarity. Style suggestions are welcome but secondary. Refer to the Tidyteam code review principles and Google’s Code Review Developer Guide for further guidance.
  • Be respectful and collegial. Code review is a learning opportunity for both the reviewer and the author. The goal is to make all of our work better, not to criticize.
